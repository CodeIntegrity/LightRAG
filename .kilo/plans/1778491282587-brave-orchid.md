- `RawNodeType` has `x`, `y`, `size`, `degree` fields; `RawGraph` manages nodeIdMap/edgeIdMap
- Graph store pattern: `useSettingsStore.use.fieldName()` for reading, `useSettingsStore.setState(...)` for writing
- Test pattern: `vi.mock('@/api/lightrag', ...)`, `vi.mock('react-i18next', ...)`, `vitest` with bun

## P0: Node Drag Implementation

### Problem
`enableNodeDrag` exists as a UI toggle but is dead — GraphControl never reads it, no drag events registered.

### Plan

#### 0.1 Add `setEnableNodeDrag` setter to settings store interface + implementation
**File:** `lightrag_webui/src/stores/settings.ts`
- Add `setEnableNodeDrag: (enable: boolean) => void` to `SettingsState` interface
- Add implementation: `setEnableNodeDrag: (enable: boolean) => set({ enableNodeDrag: enable })`
- Bump persist version to 26 (no migration needed — field already exists)

#### 0.2 Add `updateNodePosition` method to graph store
**File:** `lightrag_webui/src/stores/graph.ts`
- Add `updateNodePosition: (nodeId: string, x: number, y: number) => void` to `GraphState`
- Implementation: update `sigmaGraph` node attributes `x`/`y`, then sync to `rawGraph.nodes[nodeIndex].x/.y`
- This method does NOT increment `graphDataVersion` (position changes are view-only, not data changes)

#### 0.3 Wire drag events in GraphControl
**File:** `lightrag_webui/src/components/graph/GraphControl.tsx`
- Read `enableNodeDrag` from `useSettingsStore`
- When `enableNodeDrag` is truthy:
  - Register `downNode` event: capture node being dragged, record starting position
  - Register `moveBody` or `mousemovebody` event: if dragging, update node x/y via `updateNodePosition` and call `sigma.refresh()` 
  - Register `upNode` or `mouseup` event: release drag state, final position sync
- When `enableNodeDrag` is falsy: do not register drag events (existing behavior preserved)
- Use `useRef` for drag state (dragging node ID, start position) — NOT component state (avoids re-renders)
- Add `enableNodeDrag` to the `setSettings({...})` call to pass `nodeReducer` the drag state context if needed
- Critical: when `enableNodeDrag` changes at runtime, re-register events (add `enableNodeDrag` to the events registration useEffect dependency)

#### 0.4 Add drag conflict avoidance
- In `clickNode` handler: check if drag occurred (delta > threshold); if so, skip click
- In `enterNode`/`leaveNode`: skip hover effects while dragging
- Use `isButtonPressed` existing helper to gate drag start to left-click only

#### 0.5 Tests
**File:** `lightrag_webui/src/components/graph/GraphControl.test.tsx` (create)

Test cases:
- `enableNodeDrag=false`: drag events NOT registered, `downNode`/`moveBody` handlers absent
- `enableNodeDrag=true`: drag events registered, `downNode` handler present
- Drag updates node positions: mock sigma graph, simulate drag, verify `updateNodePosition` called
- Drag-to-click threshold: small movement does NOT suppress click; large movement DOES suppress click

### Verification
- [ ] Toggle enableNodeDrag on/off in Settings → nodes become draggable/not draggable
- [ ] Drag a node → position changes visually, stays after release
- [ ] Drag, then search, select, expand/prune → no regressions
- [ ] `cd lightrag_webui && bun test GraphControl.test.tsx` passes

## P2: View State Persistence

### Problem
Node positions initialized randomly every load; drag/layout work lost on refresh.

### Plan

#### 2.1 Create persistence utility module
**File:** `lightrag_webui/src/utils/graphViewPersistence.ts` (create)

```ts
export interface GraphViewSnapshot {
  workspace: string
  queryLabel: string
  layoutType: string
  layoutParams: Record<string, number>
  nodePositions: Record<string, { x: number; y: number }>
  cameraState?: { x: number; y: number; ratio: number }
  savedAt: number
}

export function buildPersistenceKey(workspace: string, queryLabel: string): string
export function saveGraphView(snapshot: GraphViewSnapshot): void
export function loadGraphView(workspace: string, queryLabel: string): GraphViewSnapshot | null
export function clearGraphView(workspace: string, queryLabel: string): void
export function clearAllGraphViews(): void
```

- Storage: `localStorage` with key prefix `graph-view-{workspace}-{queryLabel}`
- Serialize with JSON
- Max stored snapshots: 50 (LRU eviction)
- Validate on load: check timestamp freshness, data integrity

#### 2.2 Integrate save: capture positions before navigation/unload
**File:** `lightrag_webui/src/hooks/useLightragGraph.tsx`
- Add `useEffect` with `beforeunload` listener
- On unload: iterate `sigmaGraph` nodes, collect x/y/color/size, save via `saveGraphView()`
- Use `currentWorkspace` + `queryLabel` for the key
- Also save current layout type from settings

**File:** `lightrag_webui/src/components/graph/GraphControl.tsx`
- Save camera state (x, y, ratio) periodically (debounced) during drag/move

#### 2.3 Integrate restore: use saved positions instead of random init
**File:** `lightrag_webui/src/hooks/useLightragGraph.tsx`
- In `createSigmaGraph()`: before using `seedrandom()`, check `loadGraphView()`
- If snapshot exists for this workspace + query label:
  - Use saved positions for nodes that exist in snapshot
  - For new nodes (not in snapshot): use `seedrandom()` fallback + place near center of existing nodes
- After graph is set in store: restore camera state via `sigmaInstance.getCamera().setState(...)`

#### 2.4 Persist layout type and params
**File:** `lightrag_webui/src/stores/settings.ts`
- Layout type (`graphLayoutType`) and params are already in settings store which is persisted via zustand/persist
- Ensure layout type field exists: add `graphLayoutType: string` (default: `'Force Atlas'`)
- Add setter: `setGraphLayoutType(type: string)`
- This way layout preferences survive refresh naturally

#### 2.5 Tests
**File:** `lightrag_webui/src/utils/graphViewPersistence.test.ts` (create)

Test cases:
- `buildPersistenceKey` returns deterministic keys for same workspace+label
- `saveGraphView` + `loadGraphView` round-trip correctly
- `loadGraphView` returns null for non-existent key
- Max 50 snapshots, LRU eviction
- Node position restore: snapshot positions used instead of random
- New nodes (not in snapshot) get reasonable positions

### Verification
- [ ] Drag nodes → refresh page → positions restored
- [ ] Switch workspace or query label → different positions (no cross-contamination)
- [ ] Change layout type → refresh → layout type preserved
- [ ] New nodes added after persistence → positioned reasonably (not all at origin)
- [ ] `bun test graphViewPersistence.test.ts` passes

## Risk Mitigation

1. **Drag vs Click conflict**: Use position delta threshold (>5px → drag, ≤5px → click). Verified in P0 test.
2. **Persistence key collision**: Use `workspace + queryLabel` as key; different queries have different keys. Verified in P2 test.
3. **Large graph performance**: Persistence save iterates all nodes — use `requestIdleCallback` for large graphs. Layers disabled above threshold.
4. **@sigma/layer-webgl unavailable**: Fall back to sigma's `nodeReducer` pattern for visual highlights.
5. **Store schema version**: Bump settings persist version from 25 → 26 with migration for new layout param defaults.
