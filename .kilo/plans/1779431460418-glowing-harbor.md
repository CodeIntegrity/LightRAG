# Fix Missing API Implementations

## Summary

Fix two critical issues found in API audit:

1. **`get_graph_entity_types()`** — method referenced by `GET /graph/entity-type/list` route but not implemented in LightRAG core or any graph backend
2. **`GET /documents/scan-progress`** — endpoint called by frontend (`getDocumentsScanProgress()`) but no route handler exists

Both must be implemented in a way that minimizes merge conflicts with upstream (`HKUDS/LightRAG`).

---

## Issue 1: `get_graph_entity_types()` Missing

### Root Cause

`lightrag/api/routers/graph_routes.py:487` calls `rag.get_graph_entity_types()`, but:
- `LightRAG` has no such method (unlike `get_graph_labels` at `lightrag.py:1284`)
- `BaseGraphStorage` has no `get_all_entity_types()` abstract method (unlike `get_all_labels` at `base.py:698`)
- None of the 7 graph storage backends implement `get_all_entity_types`

### What the Test Expects

`tests/test_batch_graph_operations.py:474-512` expects:
- Returns `list[str]` of distinct entity type strings (e.g., `["CONCEPT", "TOPIC"]`)
- Triggers implicit `initialize_storages()` when storages are in `CREATED` state
- Frontend return type: `Promise<string[]>` (same as `getGraphLabels`)

### What to Implement

**A) `BaseGraphStorage.get_all_entity_types()` abstract method** (`lightrag/base.py`)

Add after `get_all_labels` (line ~704):

```python
@abstractmethod
async def get_all_entity_types(self) -> list[str]:
    """Get all distinct entity types in the graph.
    Returns a sorted list of unique entity_type values from all nodes.
    """
```

**B) Backend implementations** — 7 files, each following its `get_all_labels` pattern:

| Backend | File | Approach |
|---------|------|----------|
| NetworkX | `kg/networkx_impl.py` | Iterate `graph.nodes(data=True)`, collect `data.get("entity_type")`, sort unique |
| Neo4j | `kg/neo4j_impl.py` | Use `entity_type` parameter approach (line 1628 reference) |
| PostgreSQL | `kg/postgres_impl.py` | Similar SQL pattern to `get_all_labels` at line 6166 |
| MongoDB | `kg/mongo_impl.py` | Same pattern, different collection fields |
| OpenSearch | `kg/opensearch_impl.py` | Same pattern as labels at line 1983 |
| Nebula | `kg/nebula_impl.py` | nGQL query similar to labels at line 1732 |
| Memgraph | `kg/memgraph_impl.py` | Similar to NetworkX, iterate nodes |

Each implementation must:
1. Query all nodes
2. Extract `entity_type` from node data — handle both string and list forms (`entity_type` can be `str` OR `list[str]` — see `networkx_impl.py:553-557`). Use the `_extract_entity_types` pattern from `graph_workbench.py:223-231` or equivalent.
3. Return sorted unique list of non-empty entity type strings (deduplicated, sorted)

**C) `LightRAG.get_graph_entity_types()` method** (`lightrag/lightrag.py`)

Add after `get_graph_labels` (line ~1286):

```python
async def get_graph_entity_types(self):
    """Get all distinct entity types from the knowledge graph.
    
    Triggers implicit storage initialization if needed (same as test expectation).
    """
    return await self.chunk_entity_relation_graph.get_all_entity_types()
```

Note: The existing test expects implicit `initialize_storages()`. Rather than add auto-init to this single method (inconsistent with `get_graph_labels`), place the init in `LightRAG.get_graph_entity_types`.

**Expected merge impact**: These changes are additive (new methods only), no existing code modified. Low merge conflict risk.

## Issue 2: `GET /documents/scan-progress` Missing Route

### Root Cause

`lightrag_webui/src/api/lightrag.ts:1048-1051` calls `GET /documents/scan-progress` expecting:

```typescript
type LightragDocumentsScanProgress = {
  is_scanning: boolean
  current_file: string
  indexed_count: number
  total_files: number
  progress: number  // 0-100
}
```

But no route handler exists. The frontend currently does NOT consume this endpoint in any component yet (it uses `/health` pipeline polling instead), but the API is defined and the i18n strings are ready in 11 locales.

### What to Implement

**A) Add scan state tracking to `pipeline_status`** in `shared_storage`

Add scan-specific fields alongside existing pipeline fields:
- `scan_is_scanning: bool`
- `scan_total_files: int`
- `scan_indexed_count: int`
- `scan_current_file: str`

**B) Update `run_scanning_process()`** (`document_routes.py:2120`)

Set scan state at key points:
1. **Start**: `scan_is_scanning = True`, `scan_total_files = len(valid_files)`, `scan_indexed_count = 0`
2. **Per file** (inside `pipeline_index_files` loop): increment `scan_indexed_count`, set `scan_current_file`
3. **End/Fail**: `scan_is_scanning = False`

**C) Add `GET /documents/scan-progress` route** (`document_routes.py`)

Add to `create_document_routes()`:

```python
@router.get("/scan-progress", dependencies=[Depends(combined_auth)])
async def get_scan_progress():
    from lightrag.kg.shared_storage import get_namespace_data
    current_rag, _ = _current_runtime_objects()
    pipeline_status = await get_namespace_data(
        "pipeline_status", workspace=current_rag.workspace
    )
    total = pipeline_status.get("scan_total_files", 0) or 0
    indexed = pipeline_status.get("scan_indexed_count", 0) or 0
    return {
        "is_scanning": bool(pipeline_status.get("scan_is_scanning", False)),
        "current_file": str(pipeline_status.get("scan_current_file", "") or ""),
        "indexed_count": indexed,
        "total_files": total,
        "progress": round((indexed / total * 100) if total > 0 else 0, 1),
    }
```

**Expected merge impact**: Adds new fields to the shared `pipeline_status` dict (keys are arbitrary strings, no schema). Adds one new route handler to `create_document_routes()`. Minimal conflict risk.
